# 规则 4：每次 Codex 交互必须附带可复制文本

## 规则内容

**向 Codex 发送反馈、确认请求或等待回复时，必须在末尾附带可全选复制的中文文本**，供用户直接粘贴到 Codex 对话窗口。文本必须包含：完整上下文、确认文档路径、下一步请求。

## 为什么

用户需要频繁在 Codex 和 CC 之间传递信息。如果 CC 只在对话中输出，用户需要手动整理、翻译、补充路径后才能转发给 Codex，容易遗漏关键信息。

## 可复制文本规范（规则 5 已合并）

1. **文档路径必须包含**: 所有涉及的 review MD 文件、桌面 DOCX 文件、源码根目录的完整绝对路径
2. **能用中文处尽量用中文**: 说明、要求、步骤全部用中文；关键术语（如 CORS、FTS、YAML、API 端点名）和文件路径保留原样
3. **纯文本格式**: 不依赖 Markdown 表格或复杂格式（因为粘贴到不同聊天窗口时可能丢失格式）
4. **完整上下文**: 不能让 Codex 读完文本后还需要追问"这个文件在哪"或"你指的是哪个 Bug"
5. **明确的下一步请求**: 清楚告诉 Codex 需要做什么（如"请启动子 Agent 验证"、"请执行 Phase A"、"请反馈 CC 确认"）
6. **详实完整，附带关键代码**: 每个 Bug 必须附带：当前源码的实际代码片段（含文件路径和行号）、修复后的目标代码、修复原因、风险评估。不能只有摘要描述——Codex 必须能仅凭可复制文本就理解完整上下文并开始修复，无需反复追问
7. **完整可执行的修复计划**: 每项修复必须包含：涉及的文件（完整路径）、具体行号、当前代码（old）、目标代码（new）、验证命令。按 Phase 组织，优先级标注。Codex 读完计划就能直接执行，不需要再自己查找定位

## 可复制文本模板

```
Codex，

[背景：当前处于哪个阶段、上次交互的结论摘要]

================================================================================
本次内容
================================================================================

[每个 Bug/任务必须包含以下完整信息，不能只有摘要]

--- Bug X: [标题] ---

  文件: backend\routers\kb.py
  行号: L71
  风险: [低/中/高]
  原因: [为什么这是 Bug，影响什么功能]

  当前代码 (old):
    OBSIDIAN_VAULT = ""

  目标代码 (new):
    OBSIDIAN_VAULT = db.get_setting("obsidian_vault") or ""

  验证命令:
    python -c "import ast; ast.parse(open('backend/routers/kb.py', encoding='utf-8').read())"

--- Bug Y: [下一个任务] ---
  ...

================================================================================
读取清单
================================================================================

源码根目录：
  C:\Users\shiniyaya\Desktop\cc\B站总结工具\B站视频总结工具 -cc\

Review MD（CC 的全部分析）：
  C:\Users\shiniyaya\Desktop\总结工具修改\review\xxx.md
  C:\Users\shiniyaya\Desktop\总结工具修改\review\yyy.md

桌面 DOCX（详细报告）：
  C:\Users\shiniyaya\Desktop\xxx.docx

================================================================================
执行计划（按 Phase 组织，Codex 可直接执行）
================================================================================

Phase A（立即 — 极低风险，每项 ≤1 行）：
  A1: [任务名] — [文件]:[行号] [old] → [new] — 风险: 低
  A2: [任务名] — [文件]:[行号] [old] → [new] — 风险: 低

Phase B（优先 — 低风险，少量代码）：
  B1: [任务名] — [涉及文件列表] — 风险: 低
  B2: [任务名] — [涉及文件列表] — 风险: 中

Phase C（随后 — 中等风险）：
  C1: ...

Phase D（优化 — 需评估）：
  D1: ...

================================================================================
关键规则（必须严格遵守）
================================================================================

1. 所有文件修改必须使用 Python open(encoding="utf-8") 读写
   严禁 PowerShell Set-Content（导致 UTF-8 中文永久损坏）
   严禁 bash 管道重定向写 .js 文件（可能转码为 UTF-16 LE）

2. 每步修复后立即验证：
   Python 文件：python -c "import ast; ast.parse(open('路径', encoding='utf-8').read())"
   后端启动：python -c "import backend.main" — 确认所有 router 注册成功
   JS/HTML 文件：检查无 BOM、无 UTF-16、文件大小 > 100 字节

3. 每项修复前确认不会引入新 Bug
   如果某项修复可能引起回归，标记为"需 CC 二次确认"并暂停

4. Phase A 修复必须逐一执行，每完成一项反馈一次
   Phase B/C/D 可以在 CC 确认总体方案后批量执行

================================================================================
Codex 的执行流程
================================================================================

第1步：阅读上述所有 review MD + DOCX 文件
第2步：为每个 Phase 启动独立子 Agent，验证 CC 发现的正确性
第3步：独立验证每项修复方案的可行性
第4步：发现问题或修正意见立即反馈 CC，不盲目执行
第5步：按 Phase A → B → C → D 顺序执行修复
第6步：每完成一个 Phase，反馈 CC 确认后再进入下一 Phase

================================================================================
下一步请求
================================================================================

[按优先级排列的具体要求]
1. 请先读取 [具体文件路径] 了解完整上下文
2. 请启动 X 个子 Agent 分别验证 Phase A 的 N 项修复
3. 验证完成后反馈 CC 确认，然后开始执行 Phase A
```
