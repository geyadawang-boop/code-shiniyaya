# P0 规则：潜在优化方向提出+人工复核

## 规则内容

CC在分析开源参考文件、对比代码差异、审视架构设计时，**必须主动发现并提出潜在优化方向**，标注 `[人工复核]` 标记，写入优化计划。

很多优化方向仅靠自动化Agent无法判断价值——例如：
- "总结应该7个固定章节而非自由文本"（Agent可发现模式差异，但无法判断哪个更适合用户场景）
- "平台话术过滤可能误伤教学内容"（Agent可发现过滤规则，但需人工确认误伤范围）
- "思维导图应按时序生成4-6根节点"（Agent可发现参考实现，但节点数量需人工试效果）

## 提出格式

每个优化方向必须格式化为：
```
【参考{开源项目}】{优化方向描述}。实现: {简要方案}。[人工复核: {具体需要人工判断的内容}]
```

## 写入要求

1. 纳入优化计划记忆文件（如 bilisum-v9.0-bilinote-integration-plan.md）
2. 在DOCX报告中单独列为"潜在优化方向"章节
3. 不等待人工确认即写入记忆（标记为待复核状态）
4. 用户复核后更新状态：批准/拒绝/修改后批准

## 不可跳过的情况

- CC分析了≥2个开源参考文件 → 必须提出≥3个优化方向
- 开源文件发现BiliSum完全没有的功能 → 必须提出优化方向
- 开源文件与BiliSum对同一功能有显著不同实现 → 必须提出对比分析

## 与其他规则的关系

- 本规则不涉及代码修改——只是**分析+建议**，不需要双批准
- 发现后写入记忆即可，不等待用户即时反馈
- 用户后续复核时按 Block A 规则（修改代码需双批准）执行
