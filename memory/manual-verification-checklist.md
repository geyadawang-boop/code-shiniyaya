# 规则21: 每完成一轮修复任务即生成人工核验清单

## 规则内容

CC每完成一轮代码修复或功能添加后，必须：
1. 生成本次修复的完整人工核验清单
2. 写入此记忆文件（追加新条目，标注日期+任务）
3. 核验清单必须包含：具体步骤/期望结果/失败时如何处理
4. 用户核验后更新状态：通过/失败/部分通过（附注）

## 核验清单格式

```markdown
## [日期] 任务: {任务描述}

### 核验项 N: {核验名称}
- 状态: 待核验 | 通过 | 失败 | 部分通过
- 步骤: {具体操作步骤}
- 期望: {应该看到什么}
- 结果: {实际看到什么 — 用户填写后CC更新}
- 失败处理: {如果不符合期望该怎么办}
```

## 不可跳过的情况

- 任何涉及用户可见行为的修改（前端UI/API响应/文件输出）
- 任何涉及文件系统操作的修改（下载/删除/路径变更）
- 任何涉及数据持久化的修改（DB/JSON/ChromaDB/cache）
- 任何新功能添加
- 仅后端内部逻辑修改且不影响用户的可跳过（但需在修复记录中说明原因）

## 与其他规则的关系

- 本规则是规则17(报告→桌面)和规则20(人工复核)的补充
- 核验清单自身写入记忆即可，不需要等用户即时确认
- 用户后续反馈核验结果时，按规则15(逐任务反馈)处理

## 关联记忆
- [[human-review-optimizations]] — 优化方向人工复核
- [[reports-output-to-desktop-folder]] — 报告输出
- [[all-active-rules]]

**Why:** 多Agent自动化修复后，只有人工亲自操作软件才能发现真正的UX问题。之前多次出现CC认为修好了但用户实测发现不生效的情况（kb_dir热刷新、知识库文件位置）。系统化的人工核验清单确保每个修复都被实际测试。

**How to apply:** 每轮修复完成后立即生成清单，写入此文件。清单项编号从1开始，每轮新任务用新日期标题。用户反馈结果后更新状态。
